# 🎓 StudentMarket — что под капотом

Это маркетплейс учебных работ. Студенты могут покупать и продавать практические, лабораторные и курсовые работы. Но если вам интересно **как это всё работает изнутри** — давайте заглянем под капот.

---

## 🏗️ Архитектура

Сайт состоит из трёх частей, которые работают вместе:

### 1. Фронтенд (то, что вы видите)

Это то, с чем вы взаимодействуете в браузере:

- **index.html** — единственная HTML-страница. Никакого React, Vue или Angular. Чистый HTML с JavaScript.
- **css/style.css** — стили. Тёмная и светлая тема, адаптивная вёрстка.
- **js/script.js** — вся клиентская логика (~2600 строк). Обработка кнопок, модалок, чатов, загрузки файлов.

Всё работает через **Vanilla JavaScript** — никаких фреймворков, никаких сборщиков. Браузер загружает три файла и всё.

### 2. Бэкенд (мозг сервера)

Сервер написан на **Node.js** с **Express**. Это JavaScript, который работает на сервере, а не в браузере.

Файлы:
- **server.js** — версия для локальной разработки с SQLite. База данных хранится в одном файле `database.sqlite` прямо на диске.
- **server-pg.js** — версия для продакшена с PostgreSQL. Работает на Render в облаке.

Оба файла делают одно и то же, просто подключаются к разным базам данных. Переключение — через одну переменную окружения.

### 3. База данных (память)

Зависит от того, где запущен сайт:

**Локально (на вашем компьютере):**
- **SQLite** — вся база в одном файле. Удалите файл — удалите все данные. Удобно для разработки, потому что ничего не нужно устанавливать.

**На Render (в интернете):**
- **PostgreSQL** — полноценная реляционная база данных в облаке. Данные не пропадают при перезапуске сервера.

### 4. Авторизация

Раньше была по email/паролю. Потом переделали на **Telegram Login Widget**.

Как это работает:
1. Пользователь нажимает «Войти через Telegram»
2. Telegram спрашивает разрешение
3. Telegram отправляет данные на сайт с HMAC-подписью
4. Сервер проверяет подпись через токен бота
5. Если подпись верная — создаёт пользователя или находит существующего

Это безопаснее паролей, потому что пароль вообще не передаётся. Telegram сам подтверждает личность.

## 🔒 Безопасность

Проект был сделан с оглядкой на **OWASP Top 10** — десять самых распространённых уязвимостей веба. Вот что сделано:

| Уязвимость | Что сделано |
|---|---|
| **XSS** | Все пользовательские данные экранируются через `textContent`, а не `innerHTML`. Функция `escapeHTML()` чистит HTML-теги. |
| **SQL-инъекции** | Все SQL-запросы параметризованы (`$1`, `$2` вместо склеивания строк). |
| **Brute force** | Rate limiting — максимум 5 попыток входа за 15 минут. |
| **CSP** | Content Security Policy заголовки через Helmet. Браузер блокирует загрузку скриптов с неизвестных доменов. |
| **CORS** | Ограничен список разрешённых origin. |

## 🗃️ Схема базы данных

Вот какие таблицы есть в базе:

- **users** — пользователи (id, имя, email, пароль, баланс, isAdmin, telegram_id)
- **products** — товары (название, категория, дисциплина, цена, продавец, статус модерации)
- **purchases** — покупки (кто купил, у кого, когда, прикреплён ли файл)
- **work_files** — файлы работ (base64 данные файлов)
- **reviews** — отзывы (рейтинг 1-5, комментарий, кто оставил)
- **custom_requests** — индивидуальные заказы («нужна курсовая по теме...»)
- **chat_messages** — сообщения в чатах между покупателем и продавцом
- **notifications** — уведомления (новое сообщение, товар одобрен и т.д.)

## 🚀 Как это развёрнуто

Сайт работает на трёх бесплатных сервисах:

1. **GitHub** — хранит код. Каждый коммит = новая версия.
2. **Render** — хостит сервер. Когда код попадает в GitHub, Render автоматически пересобирает и перезапускает сервер.
3. **PostgreSQL на Render** — база данных. Хранит все данные пользователей, товаров, покупок.

Цепочка выглядит так:

```
Вы делаете commit → push на GitHub → Render видит изменения → npm install → запускает server-pg.js → сайт обновлён
```

Всё происходит автоматически. Ничего не нужно делать руками.

## 💡 Интересные детали

### Как хранятся файлы работ

Файлы (PDF, DOCX, изображения) хранятся **в базе данных** как base64-строки. Это не самый оптимальный способ (обычно файлы кладут в S3 или на диск), но для учебного проекта это проще — не нужно настраивать файловое хранилище.

### Как работают чаты

Чаты — это просто таблица `chat_messages` с полем `purchaseId`. Каждое сообщение привязано к покупке. Когда открывается чат, сервер отдаёт все сообщения для этой покупки, отсортированные по времени. Обновление каждые 3 секунды через `setInterval`. Не WebSockets, а polling — проще в реализации.

### Как работает модерация

Когда продавец создаёт товар, он получает статус `pending`. Администратор видит все `pending` товары в админ-панели и может одобрить или отклонить. Только `approved` товары видны обычным пользователям.

### Почему нет фреймворков

Это учебный проект. Цель — показать, как всё работает на низком уровне, а не спрятать всё за абстракциями React или Next.js. Когда понимаешь, как работает `fetch`, DOM, и Express «голыми руками» — потом любой фреймворк осваивается за день.

## 🛠️ Запуск локально

Если хотите поиграться:

```bash
npm install

```

Админ по умолчанию: `admin@studentmarket.ru` / `admin123`

##  Что можно улучшить

Этот проект — прототип. Вот что бы стоило доработать для продакшена:

- **Файлы вынести в S3** — хранить base64 в базе плохо для производительности
- **WebSockets для чатов** — polling каждые 3 секунды грузит сервер
- **JWT-токены** — сейчас сессия хранится в sessionStorage, это небезопасно
- **Тесты** — нет ни одного теста, ни unit, ни интеграционных
- **Пагинация** — при 1000 товарах страница будет тормозить
- **Поиск** — сейчас нет полнотекстового поиска по товарам
